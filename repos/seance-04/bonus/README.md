# Projet : créer un formulaire de contact « impossible » ou drôle

## Contexte

En développement web, un bon formulaire doit être **simple, accessible et utilisable**. Pour cet exercice bonus, vous faites volontairement l’inverse.

L’objectif est de jouer avec les **mécaniques d’interaction (HTML, CSS, JavaScript)** en créant une expérience utilisateur volontairement pénible, surprenante ou amusante.

Cet exercice s’inspire de pratiques souvent citées comme erreurs UX dans la conception de formulaires ([vidéo de référence](https://www.youtube.com/watch?v=dsY0HzfABIA)), mais utilisées ici de manière créative.

## Objectif

Créer **votre propre** formulaire de contact interactif : inventez les idées, les gags et les mécaniques. Comparez ce que vous obtenez avec les exemples du cours ou entre vous. L’idée est d’expérimenter, de vous amuser un peu et de voir jusqu’où vous pouvez pousser le concept.

Le formulaire peut devenir :

- difficile à utiliser ;
- imprévisible ;
- ou simplement drôle.

## Exemples d’idées (non exhaustif)

Vous pouvez implémenter une ou plusieurs des pistes suivantes, ou en inventer d’autres.

### Interaction « impossible »

- Bouton « Envoyer » qui rétrécit au survol
- Bouton qui se déplace pour éviter le curseur
- Champs qui changent de position
- Formulaire qui disparaît partiellement

### Interaction « drôle »

- Messages absurdes ou taquins
- Animations exagérées (shake, bounce, etc.)
- Faux retours utilisateur
- Compteur inutile (par ex. « niveau de motivation »)

### Interaction expérimentale

- Validation impossible à satisfaire
- Champs qui se remplissent tout seuls
- Inversion des libellés
- Champ qui change de type (texte → mot de passe, etc.)

## Contraintes techniques

- Utiliser du **HTML** (structure), du **CSS** (animations, transitions) et du **JavaScript** (interactions dynamiques).
- Le formulaire doit contenir au minimum :
  - nom ;
  - email ;
  - message ;
  - bouton « Envoyer ».

## Contraintes UX (volontairement violées)

Vous devez volontairement :

- rendre certaines actions difficiles ;
- perturber l’utilisateur ;
- casser les conventions habituelles.

En revanche :

- le code doit rester lisible et structuré ;
- le projet doit être exécutable dans le navigateur (même si l’expérience est frustrante).

## Livrables

- Code source (HTML / CSS / JS).
- Une courte explication (environ 5 à 10 lignes) :
  - quelles mécaniques avez-vous utilisées ?
  - pourquoi ces choix ?

## Évaluation (indicative)

| Critère | Poids suggéré |
| -------- | --------------- |
| Créativité | fort |
| Maîtrise technique | fort |
| Originalité des interactions | moyen à fort |
| Qualité du code | moyen à fort |

## Bonus possibles

- Ajouter un mode « normal » versus « impossible ».
- Ajouter un score de « frustration » utilisateur.
- Créer une petite narration (par ex. formulaire maléfique).

## Conclusion

Cet exercice vous permet de :

- mieux comprendre **ce qu’est une bonne UX en en faisant une mauvaise volontairement** ;
- manipuler le DOM et les événements ;
- expérimenter avec créativité.

Amusez-vous : plus c’est frustrant et malin, plus l’exercice est réussi.
